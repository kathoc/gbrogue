## 追加: ROM 128KB化・ボタンアイコン・持ち物3択縦メニュー(M11.16)

- **ROMを64KB→128KB**(Makefile -Wm-yo4→-Wm-yo8)。データバンクの余裕
  確保が目的。**HOMEコードは固定の先頭32KBのままで、この拡張では増えない**
  点に注意(HOME逼迫は別問題)。
- **ボタンアイコン(A/B/START/SELECT)を文字列に埋め込み**: samples の
  4枚のPNGを1bpp化(本体=インク、文字=型抜き)し、かなと同じ全角グリフ
  機構(コード>=0x80 を8x8タイルで描画、MISAKIテーブル)に6タイル
  (A/B/STARTの左右/SELECTの左右)として焼き込み。gen_lang.py が
  BUTTON_GLYPHS を harvest に強制追加し、PUAセンチネル文字(0xE000..)経由で
  ヒント/説明書の文字列に f"{BTN_A}" 等で挿入。ハーネスは lang_map.py の
  KANA でボタンコードを A/B/[/]/{/} にデコード(可読化)。対象: PACK_HINT,
  MENU_HINT, HINT_DESCEND/CLIMB/CANCEL, HELP4/5/8-12, HELP_HINT, ACT_HINT。
  ※世界ステータス行は狭いので HINT_DESCEND/CLIMB はコロン省略で1セル節約。
- **持ち物の操作選択を縦メニュー化**: アイテム選択→A で説明欄の位置に
  「> どうし / おく / やめる」の3択(行14-16)。**上下で選択・Aで決定**、
  B=もどる(一覧へ)、SELECT=とじる(持ち物を閉じる)。ヒント行(ACT_HINT)は
  「A:けってい B:もどる SELECT:とじる」(アイコン付き)。ui_inv.c の
  M_ACTION を統合(A/Bを共通末尾へ)、%3/%n 除算を分岐化して HOME 節約。
- **HOME逼迫の解消**: 上記UIで HOME(先頭32KB)がオーバー。装備統合等で
  main は収めたが、デバッグROMは -DGBR_DEBUG_KIT の**テストキット構築が
  HOMEに載って0x8000へ溢れた**。対策として**デバッグ品目のROM内構築を廃止**
  し、verify_m7/m9 が起動後に g_pack へ直接注入(gbtest.GB.inject_debug_kit、
  DEBUG_KIT に手裏剣も含む)。g_food の短縮ヒューズだけ ROM に残す。
  結果、dbg=main 相当となり両ROMとも警告0・オーバーフロー0。
- 検証: 全11本グリーン。ボタンアイコンはスクショで実描画を確認。

## 追加: ゲームボーイカラー対応(M10.6)

- カートは `-Wm-yc` の **GBC両対応**(DMG では従来の白基調のまま)。
- 起動時に `_cpu == CGB_TYPE` で実機判定(`g_is_gbc`)。
- GBC のときだけ**黒基調のデザインテーマ**を適用(単純な輝度反転ではない):
  - BGパレット4本: 文字/床=生成りの白、壁+ドア=琥珀、アイテム=アクア、
    階段=緑。背景はすべて青みの墨色 RGB(1,1,3)。
  - OBJパレット2本: プレイヤー=白緑、モンスター=橙赤。
  - タイル→パレットの分類は `attr_for_tile()`(グリフ/GFXスロットで判定)し、
    dirty 行のフラッシュ時に VRAM バンク1 の属性マップへミラー書込。
- 検証: `verify_gbc.py` が CGB ブートで「平均輝度が低い(黒基調)」
  「明るいインク画素あり」「複数色相あり」「最暗色が純黒でない(反転でない
  設計配色)」を画素サンプリングで確認。DMG 動作はヘッダの CGB フラグを
  外したコピーROM(チェックサム再計算)で従来の白基調を確認。
- 注意: PyBoy は CGB フラグ付き ROM を強制的に CGB ブートするため、
  既存テストも以後 CGB タイミングで走る(モーダル遷移は属性書込で
  従来比約2倍のフラッシュ時間 → テストは固定待ちでなく画面述語待ち
  `wait_screen`/`press_until`/カーソル閉ループに移行済み)。

## 追加: 装備自動付替+ダッシュ高速化(M11.15)

- **装備の自動付け替え**: 防具も武器と同様、装備中に別の物を装備すると
  **自動で外して付け替え**(「もう着ている」拒否を廃止)。呪われた装備が
  詰まっている場合のみ失敗(呪い発覚)。inv_equip の武器/防具ブランチを
  共通ヘルパ equip_swap に統合(HOME も約137B 回収)。指輪は2枠のまま。
- **Bダッシュの敵あり時の遅さを修正**: 原因は追跡マップ build_distmap
  (BFS)が1回で約6-7フレームかかり、**ダッシュ中は毎ステップ再構築**
  (プレイヤーが毎歩動く=キャッシュミス)していたこと。敵が1体でも起きて
  いれば 3F/step→10-12F/step に悪化していた。**ダッシュ中はBFSを4ステップに
  1回に間引く**(mons_dash、dm_dash/dm_skip)。ダッシュは敵が見えた瞬間に
  止まる=敵は画面外なので数タイル古い目標でも不可視。結果 [11,4,4,4]、
  定常4F/step(敵なしと同等)に。ダッシュ中は status_update も末尾で1回に。
- 実測: 眠っている敵は無処理(3F/step)、起きている敵の数より build_distmap
  の1回実行が支配的だった(1体でも8体でも同程度に遅かった)。
- **ダッシュ終端の壁ドン抑制**: ダッシュが壁に突き当たって終わるとき、
  従来は壁ドン演出(SFX_BUMP + 画面バウンス3フレーム)が鳴っていた。
  ダッシュ中(anim_skip)は wall_bump を即 return し、**壁のすぐ隣で静かに
  停止**する(try_move は従来どおり0を返すのでダッシュはそのセルで停止)。
  通常移動での壁ドンは従来どおり。
- **日本語モンスター名を英字頭文字に整合**(MNAME_JA): マップ記号は日英とも
  常に英大文字(`'A'+kind`)なので、頭カナが記号と一致するとイメージしやすい。
  変更: B コウモリ→バット、F ハエトリソウ→フライトラップ、
  R ガラガラヘビ→ラトルスネーク、S ヘビ→スネーク、U くろユニコーン→
  ユニコーン、V バンパイア→ヴァンパイア(K は既にチョウゲンボウ→
  ケストレル済)。P ファントム(Ph=F音)/W レイス(ワ音のカタカナ表記

### バンク構成(M10.9〜)

- MBC1 の上限は **ROM 2MB(+RAM 8KB)または ROM 512KB(+RAM 32KB)**の
  モード選択。32KB は当プロジェクトの旧ビルド設定にすぎなかった。
- 現構成: `-Wm-yo4`(64KB)。HOME(0x0000-0x7FFF 線形、バンク1常駐)に
  コード+ホットデータ、**バンク2**にピクセルデータ
  (8x8アトラス/GFXアトラス/FONT4、`#pragma bank 2`)。
- HOME コードが 0x4000 を跨ぐため **ROM 実行中のバンク切替は不可**
  (NONBANKED も _HOME が 0x4000 超に置かれ効かない — 実際に踏んだ)。
  バンク越しの読み出しは **WRAM トランポリン**(farcopy.c、手書き sm83
  40バイトを WRAM に置いて実行)経由で行う。
- 文字⇔タイル対応表(TILE8_CHARS 等)は HOME に残す(ホット参照+
  ヘッドレスハーネスが ROM から直接読むため)。

## 既知の教訓

- **文字列リテラルのネスト三項演算子は使わない**(`c ? "A" : d ? "B" : "C"`)。
  SDCC/sm83 がレジスタ圧次第でゴミポインタを返すコードを生成する
  (M10.7 で実際に踏んだ: 空腹タグが化けて表示)。if/else で書く。
- **PyBoy は CGB フラグ付き ROM に cgb=False を渡しても CGB ブートROMで
  起動する**(A=0x11 なのに機体は DMG という実機に無い半端状態)。
  ハーネスは cgb=True を既定にし、DMG 検証はヘッダの CGB フラグを外した
  コピーROM(チェックサム再計算)で行う。
- **Makefile にヘッダ依存を入れる**(`%.o: %.c $(HDRS)`)。無いとヘッダの
  定数変更が一部オブジェクトにだけ反映され、モジュール間不整合で
  極めて追いにくい壊れ方をする(T4_BASE 変更で実際に発生)。
- **メッセージ組み立てバッファは 32B 以上**。20桁画面でも合成名
  ("You killed the rattlesnake"=27B、"L1 HP120/120 St16 AC8"=22B)が
  溢れてスタック破壊 → 不可解なUIハングになる(M9 で実際に踏んだ)。
  render 側は 20 桁で切り詰めるので大きめに取って安全。
- UI 画面を閉じたら `msg_refresh()` で直近メッセージを復元する。
- game.c に描画呼び出しを増やさない(SDCC 配置問題、前プロジェクトの教訓)。
